# Описание проекта и окружения

## Цель проекта

Универсальный harness-движок для аналитической работы с базами данных (Greenplum / PostgreSQL-совместимые).
Решения должны быть **максимально универсальными** — без жёсткой привязки к конкретным схемам, таблицам или бизнес-логике конкретного заказчика.

## Технический стек

- **LLM**: GigaChat (API от Сбера)
- **Оркестрация**: LangGraph (граф нод)
- **БД**: Greenplum (PostgreSQL-совместимая MPP СУБД)
- **Запуск**: Jupyter Notebook (`agent.ipynb`) — основной способ запуска; CLI (`cli/interface.py`) официально поддерживается как дополнительный интерфейс
- **Тесты**: pytest (`tests/`)

## Жёсткие ограничения

### Задержка между запросами к LLM
Между любыми двумя вызовами LLM **обязательна пауза не менее 5 секунд** — это ограничение API GigaChat.
При добавлении новых LLM-вызовов проверяй, что задержка соблюдается (через `time.sleep(5)` или аналогичный механизм в цепочке графа).

### Контекст GigaChat
GigaChat плохо справляется с большим контекстом. Правило:
- Если изменение требует расширения промпта существующей ноды более чем на ~30% — **создай новую ноду** с отдельным вызовом LLM вместо раздувания текущего промпта.
- Одна нода = одна чётко ограниченная задача. Промпты должны быть компактными.

## Структура проекта

```
agent.ipynb          # Точка входа — запуск только через Jupyter
core/
  join_analysis.py   # Анализ JOIN-кандидатов: scoring, ranking, composite keys
  schema_loader.py   # Загрузка каталога таблиц из CSV
graph/
  nodes/
    explorer.py      # Ноды: table_explorer, column_selector
    sql_pipeline.py  # Ноды: sql_planner, sql_writer, sql_validator
  state.py           # AgentState — типизированное состояние графа
tools/
  schema_tools.py    # Инструменты для LLM: поиск таблиц, колонок, PK
tests/
  test_join_analysis.py
data_for_agent/      # CSV-каталог: tables_list.csv, attr_list.csv
```

## Архитектурные принципы

- **Граф нод**: каждая нода отвечает за один шаг (classify → explore → select columns → plan SQL → write SQL)
- **Состояние через AgentState**: ноды читают и пишут в общее состояние, не вызывают друг друга напрямую
- **LLM только там, где нужно**: детерминированная логика (scoring, matching, валидация) — в Python, не в LLM-промптах
- **Безопасность JOIN**: агент никогда не должен генерировать SQL, который множит строки; unsafe-join → DISTINCT ON или CTE с GROUP BY

## Правила для агентов при разработке

1. **Универсальность прежде всего** — никаких хардкодов схем, имён таблиц, бизнес-терминов в коде
2. **Новая логика → новая нода** — не раздувай промпты существующих нод
3. **5 секунд между LLM-вызовами** — проверяй при добавлении любого нового вызова
4. **Jupyter + CLI** — оба интерфейса официально поддерживаются; новая функциональность должна быть доступна в обоих
5. **Тесты обязательны** — любое изменение `core/join_analysis.py` покрывается тестами в `tests/test_join_analysis.py`
6. **Алиасы в SQL только на английском** — GigaChat иногда генерирует кириллицу, sql_writer должен это контролировать

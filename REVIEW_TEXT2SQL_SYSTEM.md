# Ревью Text2SQL LLM-системы (db-agent-harness)

## Общая оценка: 7.5 / 10

Система представляет собой зрелый, продуманный Text2SQL агент production-уровня с сильной архитектурой, но имеет ряд значимых пробелов, которые снижают надёжность и качество генерации SQL.

---

## Сильные стороны (что сделано хорошо)

### 1. Архитектура графа (9/10)
- LangGraph с 6 узлами (planner → table_explorer → executor ⟷ sql_validator ⟷ corrector → summarizer) — грамотная декомпозиция
- Conditional routing с защитами: MAX_GRAPH_ITERATIONS=15, wall-clock timeout 300s
- Чёткое разделение ответственности между узлами

### 2. JOIN-безопасность (9/10)
- **Лучшая часть системы.** Трёхуровневая защита от row explosion:
  - Pre-validation: проверка уникальности ключей через CSV-метаданные / DB fallback
  - Multiplication factor estimation с порогами (>2.0 = hard block)
  - Post-execution detection: анализ результата после выполнения
- Стратегии исправления дублей (WHERE-фильтр / GROUP BY / DISTINCT) с обязательной диагностикой причины
- FK-паттерн обнаружение (`{table}_id`) в table_explorer

### 3. Контекст и разведка таблиц (8/10)
- Автоматическая подгрузка структуры и семплов 10 строк из DB
- CSV-справочник с метаданными (PK, unique_perc, описания колонок)
- TTL-кэш семплов (10 мин)
- JOIN-анализ между найденными таблицами с конкретными рекомендациями

### 4. Коррекция ошибок (8/10)
- Эскалация стратегий по номеру попытки (3 retry)
- Специализированные стратегии по типу ошибки
- Long-term memory: сохранение последних 20 correction examples
- Конкретные инструкции для каждого типа ошибки в промпте

### 5. Валидация SQL (8/10)
- EXPLAIN-проверка синтаксиса
- Детекция mode (READ/WRITE/DDL) включая CTE
- Поддержка implicit JOINs, CROSS JOINs, subquery aliases
- Sanity checks результата (вопрос "сколько" + много строк → пропущена агрегация)

### 6. Безопасность (7/10)
- Identifier validation против SQL injection
- Path traversal protection
- Statement timeout 300s
- Auto-LIMIT на SELECT

## Слабые стороны и замечания

### КРИТИЧЕСКИЕ (влияют на качество SQL)

#### 1. Нет few-shot примеров SQL в промпте executor'а
**Проблема:** Executor получает правила и чеклист, но **ни одного полного примера** сложного SQL-запроса с JOIN, подзапросом, агрегацией. GigaChat (как и любая LLM) значительно лучше генерирует SQL по примерам, чем по абстрактным правилам.

**Рекомендация:** Добавить 3-5 few-shot примеров в system prompt executor'а:
- Simple SELECT с фильтром и агрегацией
- JOIN с проверкой уникальности ключа
- Подзапрос с DISTINCT ON для дедупликации
- GROUP BY с HAVING
- Пример с COALESCE для NULL-колонок

#### 2. Отсутствие семантического поиска таблиц
**Проблема:** Поиск таблиц (`_get_schema_context`) работает по точному совпадению ключевых слов (case-insensitive substring match). Пользователь спрашивает "покажи отток клиентов", а таблица называется `dm.churn_rate` — совпадения нет.

**Рекомендация:** Добавить embedding-based поиск или хотя бы fuzzy matching + маппинг синонимов (русский → английский термин).

#### 3. Planner не получает обратной связи от executor'а при неудаче плана
**Проблема:** Если плоха сама стратегия плана (выбрана не та таблица, неправильная логика), corrector пытается чинить SQL, но **replanning не происходит**. После 3 retry на шаге — шаг пропускается, а не перепланируется.

**Рекомендация:** Добавить механизм replanning: если corrector исчерпал попытки на критическом шаге, вернуться к planner с контекстом ошибки.

#### 4. Budget trimming может обрезать критический контекст
**Проблема:** `_trim_to_budget` обрезает секцию РАЗВЕДКА ТАБЛИЦ первой, но эта секция содержит **семплы данных и JOIN-анализ** — критическую информацию для корректного SQL. Обрезка может привести к генерации SQL без знания формата данных.

**Рекомендация:** Приоритизировать обрезку: сначала сокращать семплы (оставляя 3 строки вместо 10), затем убирать описания колонок для неосновных таблиц, и только потом — основную таблицу.

### ЗНАЧИМЫЕ (влияют на надёжность)

#### 5. Единственный LLM-вызов для парсинга tool_call (нет structured output)
**Проблема:** Система парсит JSON из free-text ответа LLM с regex и эвристиками (`_parse_tool_call`, `_extract_json_objects`, `_clean_llm_json`). Это хрупко — GigaChat часто ломает JSON. Retry с "форматировщиком" — это лишний LLM-вызов (5+ сек).

**Рекомендация:** Если GigaChat поддерживает function calling / structured output — использовать его. Если нет — усилить парсер: добавить json-repair библиотеку (`json_repair`).

#### 6. Нет валидации семантической корректности SQL
**Проблема:** Валидатор проверяет синтаксис (EXPLAIN) и JOIN-кардинальность, но **не проверяет семантику**:
- Запрос "сумма продаж за январь" → SQL с `WHERE date >= '2024-01-01'` без верхней границы (вернёт все данные с января)
- Запрос "количество уникальных клиентов" → SQL с `COUNT(*)` вместо `COUNT(DISTINCT client_id)`

**Рекомендация:** Добавить семантический валидатор — отдельный LLM-вызов для проверки соответствия SQL запросу пользователя (lightweight, temperature=0).

#### 7. Summarizer с temperature=0.5 может галлюцинировать
**Проблема:** Высокая температура для summarizer'а может привести к неточной интерпретации данных — LLM может "додумать" бизнес-выводы, которых нет в данных.

**Рекомендация:** Снизить temperature до 0.2-0.3. Summarizer должен быть точным, а не креативным.

#### 8. Нет кэширования LLM-ответов для идентичных запросов
**Проблема:** Rate limit 5 секунд между запросами + до 5 retry на ошибку → один шаг может занять 25+ секунд. При повторном запросе пользователя с тем же вопросом — всё заново.

**Рекомендация:** Добавить LRU-кэш для planner (одинаковый запрос + схема = одинаковый план).

### МИНОРНЫЕ (качество кода и удобство)

#### 9. Corrector не передаёт все context таблиц при row explosion
**Проблема:** При row explosion corrector получает `join_risk_info`, но не всегда получает полный `tables_context` (может быть обрезан budget trimming). Без семплов данных LLM не может определить причину дублей.

#### 10. Отсутствие метрик качества генерации
**Проблема:** Есть audit log SQL-выполнений, но нет метрик:
- % успешных SQL с первой попытки
- Среднее количество retry
- % row explosion ошибок
- Распределение ошибок по типам

#### 11. Однопроходная memory без приоритизации
**Проблема:** `correction_examples` хранит последние 20 примеров без учёта частоты/важности. Редкая ошибка, исправленная один раз, вытесняет часто повторяющийся паттерн.

#### 12. Тесты покрывают валидатор, но не промпты и граф
**Проблема:** 6 тест-файлов покрывают sql_validator, schema_loader, database, memory, path_safety, parse_tool_call. Но нет тестов для:
- Промптов (regression testing формулировок)
- Графа в целом (end-to-end с мок-LLM)
- Парсинга плана
- Budget trimming логики
